## Phase 1: MVP — ローカル対戦（人間 vs 人間、1台）

目的: ルールが実際に面白いかを検証できる最小のプレイアブルを作る。

### スコープ
- ルールエンジン（`Assets/Scripts/Core/` 一式）
  - ステートマシン（全フェーズ遷移）
  - 属性三すくみ判定
  - カード効果15種（CARDS.md 3-4章。対象選択が必要な効果もUIで実プレイ可能）
  - 得点計算
  - seed注入可能な乱数
- Unityフィールド描画（的16個・貫通式・ショットごとにランダム配置、壁グリッド・発射円1〜6・ボール。プレハブ+実行時生成）
- スリングショット操作（引っ張り→発射→物理→接触判定）
- DOMオーバーレイUI（カード選択・壁配置・スコア・フェーズ案内・履歴ログ）
- デバイス受け渡しによる2人ローカル対戦（カード秘匿画面）
- デッキ選択画面（15種から5枚）
- リザルト画面

### スコープ外（やらないこと）
- CPU対戦、オンライン機能、アカウント、演出のリッチ化、サウンド、チュートリアル

### 完了条件
- [x] 2人が1台で5ラウンド（10ショット）を最後まで遊べる（デッキ選択→対戦→リザルト→もう一度遊ぶ、を通しで確認済み）
- [x] `Core/` のユニットテスト（EditMode）が全フェーズ遷移・属性判定・15効果・得点計算をカバーし、すべてパスする（48件）
- [x] PCブラウザ（横持ち16:9基準）で問題なく操作でき、スマホブラウザでもCanvas Scalerにより破綻なく表示される
- [x] ゲーム数値がすべて `GameConfig.cs` に集約されている
- [x] WebGLビルドがunityroomのサイズ上限内に収まり、実際にアップロードして動作する（人間が行う。ARCHITECTURE.md 3.3章）

## Phase 2: CPU対戦

目的: 1人で遊べるようにし、プレイテストの回転数を上げる。

### スコープ
- `Assets/Scripts/AI/` の実装（GAME_RULES.md 9章の思考仕様）
  - カード選択AI（弱: ランダム / 強: 履歴からの推測）
  - 壁配分・配置のヒューリスティック
  - ショットの狙いノイズ（難易度別σ）
- タイトル画面にCPU戦モードと難易度選択を追加
- CPUのショットは自動再生（アニメーションで見せる）

### 完了条件
- [ ] CPU戦を最初から最後まで1人でプレイできる（実装済み・実機での通しプレイ確認待ち）
- [ ] 難易度2段階（弱/強）が体感で区別できる（実装済み・体感確認待ち）
- [x] AIはcore/のみに依存している（game/ ui/への依存なし。`LinkShot.AI.asmdef`を`noEngineReferences: true`かつ`LinkShot.Core`のみ参照で構成し、コンパイルレベルで強制）

## Phase 3: オンライン非同期対戦

目的: コンセプトの本命である「相手の布陣に挑む」非同期対戦を実現する。

### スコープ
- Supabaseセットアップ（匿名認証・matches / match_actionsテーブル。UnityWebRequestからREST APIを叩く）
- 行動の逐次記録と復元（途中再開）
- 相手のショットのリプレイ再生（初速ベクトル記録から近似再現）
- 手番通知（ポーリング方式を第一候補とする。ARCHITECTURE.md 4章参照）
- マッチング: **【暫定】ルームコード方式（友達と対戦）から始める。ランダムマッチは後回し**

### 完了条件
- [ ] 2台の端末で1試合を非同期に完走できる
- [ ] 途中でブラウザを閉じても試合状態が復元される
- [ ] 相手のショットがリプレイとして見られる

## プレイテストで検証すべき問い（Phase 1完了後）

数値調整の前に、まず以下の設計仮説が成立しているかを確認する。

1. 逐次カードセットで、実際に「相手の傾向を読む」行動が発生するか
2. 壁カード5枚の配分に悩ましさがあるか（序盤に使い切る/温存しすぎる、の両極に偏らないか)
3. 攻撃有利の体感が実現できているか（10ショット中、何本入ると気持ちよいか）
4. 500点1個を狙う一点集中と、貫通で100点/300点をまとめて拾う面狙いのリスクリターン選択が機能しているか
5. 1試合の所要時間が適切か（目標: 【暫定】10分以内）
